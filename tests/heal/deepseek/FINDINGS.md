## Defects Found

### D1 — MEDIUM: Case-sensitive manager-path check allows MANAGER.js bypass (Attack 2f)

**Severity:** MEDIUM
**Affected:** `scripts/heal.js:101-107` (`isManagerPath`)
**Attack:** Stage a repair against `MANAGER.js` (uppercase variant).

`isManagerPath()` checks:
```js
const base = path.posix.basename(relPosix);
if (base === "manager.js" || base === "manager.cjs" || base === "manager.mjs") return true;
if (/(^|\/)manager(\.[^/]+)?\.js$/.test(relPosix)) return true;
```

Both checks are **case-sensitive**. `path.posix.basename("MANAGER.js")` returns `"MANAGER.js"`, which fails the strict equality check. The regex also lacks the `/i` flag.

On **case-insensitive filesystems** (Windows NTFS, macOS HFS+/APFS default), `MANAGER.js` is the **same file** as `manager.js`. heal classifies `MANAGER.js` as `{ repair_class: "code", kind: "executable", is_manager: false }` and stages a code repair freely — violating the "manager code is NEVER modified by heal" invariant at the staging level.

**Confirmed bypass:** heal staged a code repair against `MANAGER.js` with exit code 0, `is_manager: false`, `repair_class: "code"`.

**Impact:** While code repairs remain staged-only (not auto-applied), this bypasses the explicit prohibition on even **staging** manager changes. The staged diff captures manager.js content. A social-engineering scenario (convincing a human to apply the diff) becomes possible without heal blocking it at the staging gate.

**Fix suggestion:**
```js
// In isManagerPath, make checks case-insensitive:
const baseLower = path.posix.basename(relPosix).toLowerCase();
if (baseLower === "manager.js" || baseLower === "manager.cjs" || baseLower === "manager.mjs") return true;
if (/(^|\/)manager(\.[^/]+)?\.js$/i.test(relPosix)) return true;  // add /i flag
```

### D2 — MEDIUM: Syntactic capability allowlist evadable via String.fromCharCode (Attack 3g, 3h)

**Severity:** MEDIUM
**Affected:** `scripts/heal.js:144-168` (`EXTERNAL_CALL_PATTERNS` + `capabilityPolicyScan`)
**Attack:** Construct external-call primitives from character codes or string concatenation to evade all 8 regex patterns.

**Confirmed evasions:**

1. **fromCharCode (3g):** `String.fromCharCode(101,120,101,99)` produces `"exec"`. Used as `globalThis[e]('whoami')` — zero patterns matched. `auto_apply_eligible: true`.

2. **String concatenation (3h):** `'re'+'quire'` constructs `"require"`. Used as `globalThis[r](mod).unlinkSync(...)` — zero patterns matched. `auto_apply_eligible: true`.

The 8-pattern allowlist (`node-fs`, `node-net`, `node-child-process`, `fetch`, `process-env-or-exit`, `dynamic-require`, `eval-function`, `worker-threads`) is an improvement over the v0.1 2-pattern scan, but it remains a **syntactic regex scan** — not a structural/provenance analysis.

**Impact:** A typed repair containing obfuscated external calls would be `auto_apply-eligible`. heal itself never applies (that's the orchestrator's job), but the I3-standard evaluation profile marks the repair as auto-apply-eligible, which a downstream orchestrator may trust. This creates a path where smuggled executable capabilities in `.prompt.md` or config files could be gated incorrectly.

**Fix suggestion:**
- Add patterns for `fromCharCode` + `globalThis`/`this` property access combinations
- Detect any reference to `child_process`, `fs`, `exec`, `spawn` as a literal string even if only used as a variable value
- Long-term: move toward AST-based structural analysis rather than regex-only scanning
